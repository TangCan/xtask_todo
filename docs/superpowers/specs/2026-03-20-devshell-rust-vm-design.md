# devshell 中安装 Rust 工具链并编译/运行 Rust 程序 — 设计说明

**日期**：2026-03-20  
**范围**：在 devshell 内通过轻量隔离环境（VFS 导出 → 子进程在临时目录中执行 rustup/cargo → 同步回 VFS）提供「安装 Rust 工具链」与「编译/运行 Rust 程序」能力；严格隔离（仅该进程可访问导出与临时目录）；可选扩展为 Docker/Podman 后端。  
**状态**：设计已获用户逐节通过（§1 目标与数据流、§2 隔离机制、§3 命令面与对比、§4 错误处理与测试/文档）。

---

## 1. 目标与数据流（§1 已通过）

- **目标**：在 devshell 内提供「安装 Rust 工具链」和「编译/运行 Rust 程序」能力；执行在**严格隔离环境**中进行，仅该环境可访问当前用到的 VFS 导出与临时目录。
- **数据流**：
  1. 用户在内置命令中触发（如 `rustup install stable` 或 `cargo build`）。
  2. 将 **VFS 当前子树**（默认以当前工作目录为根）**导出**到宿主机上的一个临时目录；该目录除本流程外不可见（Linux 下 fd-only，无稳定路径）。
  3. 在**隔离环境**中启动子进程，其 **cwd** 为该导出目录（通过 fd 传入或路径）；在该目录下执行 **rustup** / **cargo** / **rustc**（宿主机已安装的二进制）。
  4. 子进程结束后，将导出目录中的**变更**（新增/修改/删除的文件）**同步回**内存中的 VFS；然后关闭 fd、删除临时目录（若仍有路径则清理），完成一次调用。
- **不做的**：不实现通用「任意宿主命令」沙箱；本阶段仅支持与 Rust 工具链相关的这条链路。

## 2. 隔离机制（§2 已通过）

### 2.1 Linux（方案 B，fd-only）

- 在 `$TMPDIR` 或 `/tmp` 下创建唯一临时目录（如 `devshell_<uuid>`），权限 `0700`。
- 立即对该目录做 **unlink**（或等价操作），使目录在父级目录中不再有名字；父进程只保留该目录的**打开 fd**（不再通过路径访问）。
- 子进程通过 **fd 作为 cwd** 启动：用 `openat(fd, ".", O_RDONLY)` 得到可执行目录 fd，或通过 `procfs` 的 `/proc/self/fd/<n>` 作为 cwd 传给 `Command`；确保子进程及其派生的 rustup/cargo/rustc 仅在该树下运行。
- 执行结束后，父进程用保留的 fd 遍历目录、将变更同步回 VFS，然后关闭 fd；目录随 fd 关闭由内核回收（无路径则无人能再打开）。

### 2.2 非 Linux（回退到方案 A）

- 仅使用**随机路径 + 0700**：在 `$TMPDIR` 或平台临时目录下创建 `devshell_<uuid>`，权限设为仅当前用户可访问；子进程 cwd 设为该路径；结束后同步并删除该目录。不保证「仅本进程可访问」，但实现简单、可移植。
- 文档中写明：严格隔离在 Linux 上为 fd-only，其他平台为尽力而为。

### 2.3 依赖与约束

- 不依赖 setuid 或 root；若平台不支持「目录 unlink 后仅 fd 可见」（如部分 Windows），则仅做方案 A。

### 2.4 轻量隔离达不到时的备选（Docker/Podman）

- **原则**：同一套「导出 VFS → 在环境中执行 rustup/cargo → 同步回 VFS」的流程不变，只替换「环境」的实现。
- **当前**：轻量隔离（Linux fd-only，非 Linux 随机路径 + 0700）。
- **若轻量隔离不够**（策略要求、审计、或平台限制导致 fd-only 不可用）：可增加**可选**的容器模式：
  - 将 VFS 子树导出到宿主机临时目录（与现在一致）。
  - 使用 **Docker 或 Podman** 启动一个容器，把该目录挂载到容器内的工作目录（如 `-v /tmp/devshell_xxx:/workspace -w /workspace`），在容器内执行 `rustup`/`cargo`（镜像内预装或挂载宿主工具链，由实现选择）。
  - 容器退出后，从该目录同步变更回 VFS，再删除临时目录。
- **触发方式**（实现时定）：例如环境变量 `DEVSHELL_ISOLATION=container`、或 devshell 配置项、或 `cargo --isolate=container` 一类参数；默认仍为轻量隔离。
- **约束**：容器模式为可选扩展；首期可只做轻量隔离，若需更强隔离可后续增加 Docker/Podman 后端，数据流与接口保持一致。

## 3. devshell 命令面（§3 已通过）

### 3.1 命令形态

- **`rustup`** — 子命令与参数原样转发给宿主机的 `rustup`（如 `rustup install stable`、`rustup default stable`）。在隔离环境中执行，工作目录为「当前 VFS 导出」的根。
- **`cargo`** — 子命令与参数原样转发给宿主机的 `cargo`（如 `cargo build`、`cargo run`、`cargo new foo`）。同样在隔离环境中执行，cwd 为导出根；编译产物、`target/`、新 crate 等会落在导出目录，结束后同步回 VFS。
- 可选：**`rustc`** — 若需要直接调用编译器，可同样以内置形式转发给宿主 `rustc`；首期可与 cargo 二选一或都支持，实现时定。

### 3.2 导出范围

- 默认：以 **VFS 当前工作目录（cwd）** 为根，导出整棵子树到临时目录；子进程看到的根即该子树（相对路径与 devshell 内一致）。
- 可选：支持 `rustup`/`cargo` 带一个路径参数表示「从某目录执行」（导出该路径对应子树）；首期可只做「当前 cwd」以简化。

### 3.3 查找宿主二进制

- 通过 `PATH` 查找 `rustup`、`cargo`（及可选 `rustc`）；若未找到，报错并提示安装（如 "rustup not found in PATH"），不执行导出。

### 3.4 与现有内置的关系

- `rustup` / `cargo`（及可选 `rustc`）为新增内置，与 `pwd`、`cd`、`todo` 等并列；不改变现有管道、重定向语义，这些命令参与管道时 stdout/stderr 与现有一致。

### 3.5 与 Docker/Podman 的区别

| 维度         | 本设计（devshell 隔离环境）                                                                 | Docker / Podman                                                                 |
|--------------|---------------------------------------------------------------------------------------------|----------------------------------------------------------------------------------|
| **边界**     | 同一台宿主机上的**进程级隔离**：临时目录 + fd-only（或随机路径），子进程仅能访问导出的 VFS 子树。 | **容器**：独立命名空间（mount/network/PID/UTS 等）、镜像层、独立根文件系统。     |
| **文件系统** | 数据来自 devshell 的 **内存 VFS**，导出到临时目录供子进程使用，结束后同步回 VFS；无镜像、无层。   | 基于**镜像层**和**卷挂载**的持久化文件系统。                                      |
| **用途**     | 在 **devshell 会话内**用宿主已安装的 rustup/cargo 操作 **同一份 VFS 内容**，且仅该次调用的进程可访问该导出。 | 构建/运行**可复现环境**、多服务编排、跨机器部署。                                 |
| **隔离强度** | 仅保证「只有本次导出的进程树能访问该临时目录」；不做网络/进程/用户等命名空间隔离。               | 强隔离：独立网络、PID、用户等；可配合 seccomp、cgroup 做资源与能力限制。          |
| **依赖**     | 依赖宿主机已安装的 rustup/cargo 和 PATH；无 daemon、无镜像拉取。                             | 需要容器运行时与镜像；Docker 需 daemon。                                         |

**一句话**：本设计是「在 devshell 里用宿主工具链、但只让这次调用看到当前 VFS」的轻量隔离；Docker/Podman 是「独立容器环境与镜像体系」，两者目标与实现都不同。若轻量隔离不足，可按 §2.4 增加可选的容器后端。

## 4. 错误处理、清理与测试/文档（§4）

### 4.1 错误处理

- **导出失败**（如无法创建临时目录、无法写入 VFS 内容）：向 stderr 报错，不启动子进程；返回非零或等效 BuiltinError。
- **宿主二进制未找到**：报错提示安装 rustup/cargo，不执行导出。
- **子进程执行失败**（如 rustup/cargo 返回非零）：将子进程的 stdout/stderr 透传到 devshell；命令整体视为失败（脚本下若 `set -e` 则终止）。
- **同步回 VFS 失败**（如冲突、权限）：报错并尽可能保留导出目录路径或 fd 便于排查；实现时可定义是否部分写回或全部放弃。

### 4.2 清理

- 无论成功与否，在单次调用结束前：关闭保留的 fd；若临时目录仍有路径（非 Linux 或 fallback），删除该目录及其内容。
- 子进程被 kill 或崩溃时，父进程仍执行清理逻辑（fd 关闭、目录删除）。

### 4.3 测试

- 单元/集成：在测试中构造 VFS（含 Cargo.toml 或空目录），调用 rustup/cargo 内置，断言导出目录存在且子进程 cwd 正确、同步后 VFS 内容符合预期；可 mock 宿主 rustup/cargo 或使用真实二进制（若 CI 具备）。
- 隔离：在 Linux 上可验证「导出目录在 unlink 后仅通过 fd 可访问」（如其他进程无法通过路径打开）。

### 4.4 文档

- README 或 devshell 文档增加「Rust 工具链」小节：说明 `rustup`/`cargo` 内置、隔离方式（fd-only / 随机路径）、导出与同步语义、与 Docker/Podman 的差异及可选容器后端；给出示例（如 `cd my_project` → `cargo build`）。
