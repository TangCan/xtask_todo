## 1. 设计目标

在不大改需求文档结构的前提下，集中补全：

- **边界与数据约定**：id、可选字段格式与长度、文件与 I/O、规模与限制。
- **错误行为汇总**：按退出码与错误类型归纳 CLI 行为，便于实现与测试查阅。
- **已知限制与未覆盖需求**：明确当前版本不包含的能力，与 §5「未来可考虑」衔接。

## 2. 变更内容摘要

### 2.1 边界、异常与数据约定（§3.1）

- **ID**：id 为正整数；0 视为非法；不存在的 id 在 complete/delete/update/show 中返回数据错误（退出码 3）。
- **可选字段**：日期 YYYY-MM-DD；优先级 low/medium/high；repeat_rule 为约定枚举或 custom:N；repeat_count 为正整数；非法格式返回参数错误（退出码 2）。
- **文件与 I/O**：默认读写 `.todo.json`；文件不存在时 list/show 等行为按当前实现约定；导入/导出时文件不可写或格式错误应给出明确错误。
- **规模与限制**：当前不承诺列表长度或导出体积上限；必要时可写「建议/合理使用范围」（如单文件数千条内）。

### 2.2 错误行为汇总（§3.2）

- 退出码：0 成功；1 一般错误；2 参数错误；3 数据错误。
- 表格：典型场景（空标题 add、非法 id complete、无效 --due-date、不存在的 id update 等）与对应退出码及错误类型。

### 2.3 已知限制与未覆盖需求（§3.3）

- 当前版本不包含：HTTP API、多用户、备份/恢复策略、并发写入保证、schema 版本迁移等。
- 与 §5「未来可考虑」一致，采用「当前不覆盖」的正面表述。
