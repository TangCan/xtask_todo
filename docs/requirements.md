# 需求说明 (Requirements)

## 1. 项目概述

- **项目名称**：xtask_todo  
- **目标**：提供可复用的待办（Todo）能力（库 + 可选入口），并通过 cargo xtask 支持开发与构建自动化。  
- **范围**：本需求覆盖「todo 领域能力」与「xtask 开发者工作流」两部分的用户故事与验收标准。

**当前实现状态**：US-T1～US-T6、US-X1～US-X4 与扩展 US-T7～US-T13、US-A1～US-A4 均已实现。CLI 入口为 `cargo xtask todo`，子命令含 add、list、show、update、complete、delete、search、stats、export、import、init-ai。**add** 支持可选参数 `--description`、`--due-date`、`--priority`、`--tags`、`--repeat-rule`；**update** 支持同上可选参数及标题；**export** 支持 JSON/CSV（由文件扩展名或 `--format json|csv` 决定）；**import** 支持 JSON 与 CSV（按扩展名识别）。重复规则支持 `daily`/`weekly`/`monthly`/`yearly`/`weekdays`、`2d`/`3w` 简写及 `custom:N`，并支持结束条件 `repeat_until`（截止日期）、`repeat_count`（剩余次数）。全局选项含 `--json`、`--dry-run`（修改类命令仅输出拟执行操作，不写入文件且不修改内存列表）；退出码为 0（成功）/1（一般）/2（参数）/3（数据）。

## 2. 用户故事与验收标准

### 2.1 Todo 领域（crates/todo）

#### US-T1：创建待办

- **用户故事**：作为用户，我希望可以创建一条待办（含标题等必要信息），以便后续跟踪与完成。
- **验收标准**：
  - 给定 任意有效标题，当 调用创建接口 时，应 返回一个可识别的待办实例（如 id 或句柄）。
  - 给定 空标题或非法输入（按产品约定），当 调用创建接口 时，应 返回明确错误或拒绝创建，且不产生有效待办。

#### US-T2：列出待办

- **用户故事**：作为用户，我希望可以按列表查看当前所有待办，以便了解整体进度。
- **验收标准**：
  - 给定 零条待办，当 请求列表 时，应 返回空列表。
  - 给定 已存在若干条待办，当 请求列表 时，应 返回包含这些待办的列表，且顺序或可筛选行为符合约定（如按创建时间）。

#### US-T3：完成待办

- **用户故事**：作为用户，我希望可以将某条待办标记为已完成，以便区分未完成与已完成。
- **验收标准**：
  - 给定 一条未完成的待办，当 将其标记为已完成 时，该待办在后续查询中应 表现为「已完成」。
  - 给定 不存在的 id 或非法 id，当 尝试标记完成 时，应 返回明确错误，且不改变其他待办状态。

#### US-T4：删除待办

- **用户故事**：作为用户，我希望可以删除一条待办，以便清理不再需要的项。
- **验收标准**：
  - 给定 一条已存在的待办，当 执行删除 时，该待办从列表中消失，后续按 id 查询应 无法再获取到该项。
  - 给定 不存在的 id，当 执行删除 时，应 返回明确错误或幂等成功，且不破坏现有数据一致性。

#### US-T5：时间戳与完成时间

- **用户故事**：作为用户，我希望系统记录任务的创建时间与完成时间，并在列表中展示，以便追踪进度与完成用时。
- **验收标准**：
  - 创建待办时，系统应 记录创建时间（如 `created_at`）；完成待办时，应 记录完成时间（如 `completed_at`）。
  - 列表展示时，应 展示创建时间（及已完成项的完成时间）；已完成项应 可展示「从创建到完成」的用时。

#### US-T6：长时间未完成提醒

- **用户故事**：作为用户，我希望在列表中对待办时间过长的未完成任务（如超过一周）有视觉区分（如不同颜色），以便优先处理。
- **验收标准**：
  - 在支持终端颜色的环境下列出待办时，创建超过约定阈值（如 7 天）且未完成的任务应 以不同颜色或样式展示。
  - 在非 TTY 或管道输出时，不输出颜色码，保持纯文本。

#### US-T7：查看单条任务（扩展，P1）

- **用户故事**：作为用户，我希望通过 `todo show <id>` 查看单条任务的完整信息，以便快速确认详情。
- **验收标准**：
  - 给定有效 id，应 输出该任务的完整字段（id、标题、创建时间、完成时间、状态等）。
  - 给定不存在的 id，应 返回明确错误信息及非 0 退出码。

#### US-T8：更新任务（扩展，P1）

- **用户故事**：作为用户，我希望通过 `todo update <id>` 修改任务的标题或其他属性，以便更正或补充信息。
- **验收标准**：
  - 给定有效 id 与合法修改内容，应 更新该任务并持久化；列表与 show 应 反映更新后内容。
  - 给定不存在的 id 或非法参数，应 返回明确错误，且不改变其他任务。

#### US-T9：任务可选属性（扩展，P1）

- **用户故事**：作为用户，我希望任务除标题外可包含描述、截止日期、优先级、标签等可选属性，以便更好分类与排序。
- **验收标准**：
  - 添加/更新任务时，应 支持可选字段：描述、截止日期、优先级（如高/中/低）、标签列表；列表与 show 应 能展示这些字段。
  - 列表应 支持按状态、优先级、标签、截止日期过滤及排序（实现范围可按优先级分阶段）。

#### US-T10：搜索任务（扩展，P1）

- **用户故事**：作为用户，我希望通过 `todo search <keyword>` 在标题、描述、标签中搜索任务，以便快速定位。
- **验收标准**：
  - 给定关键词，应 返回匹配任务列表；无匹配时返回空列表。
  - 搜索范围应 至少包含标题；若已实现描述/标签，应 一并纳入搜索。

#### US-T11：统计信息（扩展，P2）

- **用户故事**：作为用户，我希望通过 `todo stats` 查看任务总数、已完成数、未完成数等统计，以便了解整体进度。
- **验收标准**：
  - 输出应 包含至少：总任务数、未完成数、已完成数；可扩展逾期数、按优先级分布等。

#### US-T12：导入与导出（扩展，P2）

- **用户故事**：作为用户，我希望将任务导出为 JSON/CSV 或从文件导入，以便备份或迁移。
- **验收标准**：
  - `todo export <file>` 应 将当前任务列表导出为指定格式（由文件后缀或选项决定）。
  - `todo import <file>` 应 从文件导入任务，与现有数据合并或按约定策略去重/覆盖。

#### US-T13：定期重复任务（扩展，P1）

- **用户故事**：作为用户，我希望为任务设置重复规则（如每日/每周/每月），在标记完成时自动生成下一次实例，以便管理周期性待办。
- **验收标准**：
  - **定义重复规则**：添加或更新任务时，应 支持可选重复规则，至少包括：`daily`、`weekly`、`monthly`、`yearly`、`weekdays`（工作日），以及自定义间隔（如 `2d`、`3w`）；可附加结束条件（重复次数或截止日期）。
  - **自动生成新实例**：当带重复规则的任务被标记为完成时，系统应 根据规则自动创建下一次任务，并计算新任务的截止日期（若任务含截止日期）。
  - **跳过下次生成**：标记完成时支持 `--no-next` 选项，仅完成当前实例、不生成下一次任务（P2）。
  - **查看与修改**：在任务详情（`todo show <id>`）中应 展示重复规则；支持通过 `todo update <id>` 修改或取消重复规则（P1）。

### 2.2 Xtask 工作流（开发者）

#### US-X1：通过 cargo xtask 执行任务

- **用户故事**：作为开发者，我希望通过 `cargo xtask <子命令>` 执行项目内约定任务，以便统一入口、不依赖外部脚本。
- **验收标准**：
  - 在项目根目录执行 `cargo xtask --help` 时，应 输出用法说明及可用子命令列表。
  - 执行 `cargo xtask <已实现的子命令>` 时，应 执行对应任务并返回适当退出码（成功 0，失败非 0）。

#### US-X2：运行主程序（示例任务）

- **用户故事**：作为开发者，我希望通过 `cargo xtask run` 一键运行主程序（或当前约定的“运行”行为），以便快速验证。
- **验收标准**：
  - 执行 `cargo xtask run` 时，应 成功启动并执行当前定义的「run」逻辑（如运行主二进制或启动服务）。
  - 若 run 依赖未满足（如缺少配置），应 输出明确错误信息并以非 0 退出码退出。

#### US-X3：扩展新的 xtask 子命令

- **用户故事**：作为开发者，我希望在 `xtask` 中新增子命令时，仅需在代码中增加子命令定义与实现，无需改动 cargo 配置即可通过 `cargo xtask <新命令>` 调用。
- **验收标准**：
  - 在 xtask 源码中新增一个子命令（如 `build`、`release`）并实现后，执行 `cargo xtask <新命令>` 应 执行该实现。
  - `cargo xtask --help` 中应 展示新子命令的名称与说明。

#### US-X4：cargo xtask todo 任务管理

- **用户故事**：作为用户，我希望通过 `cargo xtask todo` 子命令（add/list/complete/delete）管理待办，数据持久化到本地文件，以便在多次运行间保留。
- **验收标准**：
  - `cargo xtask todo add "标题"` 应 新增一条待办并持久化；`cargo xtask todo list` 应 列出所有待办（含创建/完成时间与用时）；`cargo xtask todo complete <id>`、`cargo xtask todo delete <id>` 应 生效并持久化。
  - 数据应 保存在项目目录下的约定文件（如 `.todo.json`），重启或再次运行后仍可读写。

### 2.3 AI/CLI 集成（扩展，参考技能集成版）

以下需求便于 AI 助手或脚本可靠地调用 TODO 命令并解析结果，优先级标注供实现排序。

#### US-A1：结构化 JSON 输出（P0）

- **用户故事**：作为调用方（含 AI 助手），我希望所有子命令支持 `--json` 选项并输出结构化数据，以便程序化解析。
- **验收标准**：
  - 在子命令后附加 `--json` 时，应 输出合法 JSON，包含操作结果或错误信息；成功时包含 `status` 与 `data`，失败时包含 `status` 与 `error`（含 code、message）。
  - 非 `--json` 时保持人类可读的现有输出，行为不变。

#### US-A2：标准退出码（P1）

- **用户故事**：作为脚本或 AI，我希望命令返回一致退出码，以便判断成功与错误类型。
- **验收标准**：
  - 成功为 0；一般错误为 1；参数错误（缺少/格式错误）为 2；数据操作失败（如 id 不存在）为 3。

#### US-A3：AI 技能生成命令（P0）

- **用户故事**：作为开发者，我希望通过 `todo init-ai` 为目标 AI 助手（如 Cursor、Claude Code）生成技能/指令文件，以便 AI 学会正确调用 TODO 命令。
- **验收标准**：
  - 执行 `todo init-ai`（或 `cargo xtask todo init-ai`）并指定目标（如 `--for cursor`）时，应 在约定目录（如 `.cursor/commands/`）生成对应格式的技能文件。
  - 技能文件应 包含元数据（名称、描述、触发方式）及自然语言指令，指导 AI 解析用户意图并构造带 `--json` 的命令。

#### US-A4：模拟执行（P2）

- **用户故事**：作为用户或 AI，我希望对修改类命令（add/update/complete/delete）支持 `--dry-run`，仅展示将要执行的操作而不写数据，以便预览。
- **验收标准**：
  - 带 `--dry-run` 时，应 输出拟执行的操作说明，且不修改 `.todo.json`，也不对内存中的待办列表做增删改（仅预览）。

## 3. 非功能与约束

- **实现语言**：Rust；workspace 包含 `crates/todo` 与 `xtask`。
- **入口**：通过 `.cargo/config.toml` 的 `xtask` alias 支持 `cargo xtask`，不要求用户全局安装额外工具。
- **验收方式**：上述验收标准可通过单元测试、集成测试或手工执行验证；建议为 US-T* 编写自动化测试，US-X* 可配合脚本或 CI 验证。
- **命令稳定性**：主版本内 CLI 子命令与主要选项保持向后兼容；变更前应 提供废弃警告或文档说明。
- **文档**：提供命令用法说明（如 `--help` 及 README）；新增子命令或选项时同步更新文档与示例。
