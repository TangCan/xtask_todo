# 需求说明 (Requirements)

## 1. 项目概述

- **项目名称**：xtask_todo  
- **目标**：提供可复用的待办（Todo）能力（库 + 可选入口），并通过 cargo xtask 支持开发与构建自动化。  
- **范围**：本需求覆盖「todo 领域能力」与「xtask 开发者工作流」两部分的用户故事与验收标准。

---

## 2. 用户故事与验收标准

### 2.1 Todo 领域（crates/todo）

#### US-T1：创建待办

- **用户故事**：作为用户，我希望可以创建一条待办（含标题等必要信息），以便后续跟踪与完成。
- **验收标准**：
  - 给定 任意有效标题，当 调用创建接口 时，应 返回一个可识别的待办实例（如 id 或句柄）。
  - 给定 空标题或非法输入（按产品约定），当 调用创建接口 时，应 返回明确错误或拒绝创建，且不产生有效待办。

#### US-T2：列出待办

- **用户故事**：作为用户，我希望可以按列表查看当前所有待办，以便了解整体进度。
- **验收标准**：
  - 给定 零条待办，当 请求列表 时，应 返回空列表。
  - 给定 已存在若干条待办，当 请求列表 时，应 返回包含这些待办的列表，且顺序或可筛选行为符合约定（如按创建时间）。

#### US-T3：完成待办

- **用户故事**：作为用户，我希望可以将某条待办标记为已完成，以便区分未完成与已完成。
- **验收标准**：
  - 给定 一条未完成的待办，当 将其标记为已完成 时，该待办在后续查询中应 表现为「已完成」。
  - 给定 不存在的 id 或非法 id，当 尝试标记完成 时，应 返回明确错误，且不改变其他待办状态。

#### US-T4：删除待办

- **用户故事**：作为用户，我希望可以删除一条待办，以便清理不再需要的项。
- **验收标准**：
  - 给定 一条已存在的待办，当 执行删除 时，该待办从列表中消失，后续按 id 查询应 无法再获取到该项。
  - 给定 不存在的 id，当 执行删除 时，应 返回明确错误或幂等成功，且不破坏现有数据一致性。

---

### 2.2 Xtask 工作流（开发者）

#### US-X1：通过 cargo xtask 执行任务

- **用户故事**：作为开发者，我希望通过 `cargo xtask <子命令>` 执行项目内约定任务，以便统一入口、不依赖外部脚本。
- **验收标准**：
  - 在项目根目录执行 `cargo xtask --help` 时，应 输出用法说明及可用子命令列表。
  - 执行 `cargo xtask <已实现的子命令>` 时，应 执行对应任务并返回适当退出码（成功 0，失败非 0）。

#### US-X2：运行主程序（示例任务）

- **用户故事**：作为开发者，我希望通过 `cargo xtask run` 一键运行主程序（或当前约定的“运行”行为），以便快速验证。
- **验收标准**：
  - 执行 `cargo xtask run` 时，应 成功启动并执行当前定义的「run」逻辑（如运行主二进制或启动服务）。
  - 若 run 依赖未满足（如缺少配置），应 输出明确错误信息并以非 0 退出码退出。

#### US-X3：扩展新的 xtask 子命令

- **用户故事**：作为开发者，我希望在 `xtask` 中新增子命令时，仅需在代码中增加子命令定义与实现，无需改动 cargo 配置即可通过 `cargo xtask <新命令>` 调用。
- **验收标准**：
  - 在 xtask 源码中新增一个子命令（如 `build`、`release`）并实现后，执行 `cargo xtask <新命令>` 应 执行该实现。
  - `cargo xtask --help` 中应 展示新子命令的名称与说明。

---

## 3. 非功能与约束

- **实现语言**：Rust；workspace 包含 `crates/todo` 与 `xtask`。
- **入口**：通过 `.cargo/config.toml` 的 `xtask` alias 支持 `cargo xtask`，不要求用户全局安装额外工具。
- **验收方式**：上述验收标准可通过单元测试、集成测试或手工执行验证；建议为 US-T* 编写自动化测试，US-X* 可配合脚本或 CI 验证。
