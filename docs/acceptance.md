# 验收文档 (Acceptance)

本文档用于对 [requirements.md](./requirements.md) 中的用户故事与验收标准进行逐项验证，可作为发布或迭代结束时的签字依据。每项验收给出**验证方式**与**预期结果**，通过则在「结果」栏勾选。

---

## 1. 验收说明

- **适用范围**：当前迭代/版本对应的需求范围，以 requirements.md 为准。
- **验证方式**：自动化（测试用例名/命令）或手工步骤；若以测试为主，可注明「由 CI 执行」。
- **结果**：✅ 通过 / ❌ 未通过 / ⏸ 跳过（注明原因）。
- **签字**：验收通过后由产品或项目负责人签字及日期（可写在文末或单独附录）。

## 2. Todo 领域验收

### 2.1 US-T1：创建待办

| # | 验收标准 | 验证方式 | 预期结果 | 结果 |
|---|----------|----------|----------|------|
| T1-1 | 给定任意有效标题，调用创建接口时，应返回可识别的待办实例（如 id） | 单元/集成测试：对非空标题调用 `TodoList::create`，断言返回 `Ok(id)`，且 `list()` 中包含该项 | 返回 `Ok(TodoId)`，列表中有一条 title 匹配、id 一致的 Todo | |
| T1-2 | 给定空标题或非法输入，调用创建接口时应返回明确错误且不产生有效待办 | 测试：对空字符串（及约定非法输入）调用 `create`，断言返回 `Err`；再调用 `list()` 断言条数未增加 | 返回 `Err(TodoError::InvalidInput)`（或等价），列表长度不变 | |

**建议测试位置**：`crates/todo/tests/` 或 `crates/todo/src/` 内单元测试。

---

### 2.2 US-T2：列出待办

| # | 验收标准 | 验证方式 | 预期结果 | 结果 |
|---|----------|----------|----------|------|
| T2-1 | 给定零条待办，请求列表时应返回空列表 | 测试：新建 `TodoList` 后调用 `list()` | 返回 `Vec::new()` 或等价空列表 | |
| T2-2 | 给定已存在若干条待办，请求列表时应返回包含这些待办的列表，且顺序符合约定（如按创建时间） | 测试：依次 `create` 若干条，再 `list()`，断言数量与内容；断言顺序为创建时间升序（或文档约定） | 列表长度与 create 次数一致，顺序与设计约定一致 | |

---

### 2.3 US-T3：完成待办

| # | 验收标准 | 验证方式 | 预期结果 | 结果 |
|---|----------|----------|----------|------|
| T3-1 | 给定一条未完成的待办，将其标记为已完成时，该待办在后续查询中应表现为「已完成」 | 测试：`create` → `complete(id)` → `list()` 或 `get(id)`，断言该项 `completed == true` | 列表中该 id 对应项 `completed` 为 true | |
| T3-2 | 给定不存在的 id，尝试标记完成时应返回明确错误，且不改变其他待办状态 | 测试：`create` 一条，对不存在的 `TodoId` 调用 `complete`，断言 `Err(NotFound)`；再 `list()` 断言原有一条且仍为未完成 | 返回 `Err(TodoError::NotFound)`，原有待办状态不变 | |

---

### 2.4 US-T4：删除待办

| # | 验收标准 | 验证方式 | 预期结果 | 结果 |
|---|----------|----------|----------|------|
| T4-1 | 给定一条已存在的待办，执行删除后该待办从列表中消失，按 id 无法再获取 | 测试：`create` → 记下 id → `delete(id)` → `list()` 不包含该 id | 列表无该项，按该 id 查询为 None 或等价 | |
| T4-2 | 给定不存在的 id，执行删除时应返回明确错误或幂等成功，且不破坏现有数据一致性 | 测试：对未存在的 id 调用 `delete`，断言约定行为（Err 或 Ok）；再 `list()` 断言条数不变 | 符合 design 约定（Err 或幂等 Ok），列表内容与条数不变 | |

## 3. Xtask 工作流验收

### 3.1 US-X1：通过 cargo xtask 执行任务

| # | 验收标准 | 验证方式 | 预期结果 | 结果 |
|---|----------|----------|----------|------|
| X1-1 | 在项目根目录执行 `cargo xtask --help` 时，应输出用法说明及可用子命令列表 | 在项目根执行：`cargo xtask --help` | 输出中包含用法说明及子命令列表（如 `run`） | |
| X1-2 | 执行 `cargo xtask <已实现的子命令>` 时，应执行对应任务并返回适当退出码 | 执行 `cargo xtask run`（及已实现的其他子命令）；成功时查退出码为 0，失败场景为非 0 | 成功：退出码 0；失败：非 0 | |

---

### 3.2 US-X2：运行主程序（示例任务）

| # | 验收标准 | 验证方式 | 预期结果 | 结果 |
|---|----------|----------|----------|------|
| X2-1 | 执行 `cargo xtask run` 时，应成功启动并执行当前定义的「run」逻辑 | 在项目根执行：`cargo xtask run` | 进程正常执行并退出，无 panic；若有主二进制则主逻辑被触发 | |
| X2-2 | 若 run 依赖未满足（如缺少配置），应输出明确错误信息并以非 0 退出码退出 | 在依赖未满足环境下执行 `cargo xtask run`（若当前无此类依赖可标为 N/A 或后续补充） | stderr 有明确错误说明，退出码非 0 | |

---

### 3.3 US-X3：扩展新的 xtask 子命令

| # | 验收标准 | 验证方式 | 预期结果 | 结果 |
|---|----------|----------|----------|------|
| X3-1 | 在 xtask 源码中新增子命令并实现后，`cargo xtask <新命令>` 应执行该实现 | 在 xtask 中新增子命令（如 `build`），实现后执行 `cargo xtask build` | 新命令被正确解析并执行 | |
| X3-2 | `cargo xtask --help` 中应展示新子命令的名称与说明 | 执行 `cargo xtask --help` | 帮助文本中包含新子命令及其描述 | |

*注：若当前未新增子命令，可仅验证现有 `run` 在帮助中展示，或在下一次扩展时勾选。*
